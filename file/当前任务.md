4.20 规划：人脸识别迁移至云端计算全流程（仅规划，不改代码）
omniattend-core

规划目标
：
1. 将人脸识别从“端侧提取/判定”收敛为“云端提取/比对/判定”，Android 仅保留采集上传与提交。
2. 不破坏现有签到能力：定位、手势、密码、审核队列继续可用。
3. 在不新增复杂系统的前提下，复用现有 `CheckinTask/CheckinSubmission/FaceEmbedding/R2` 体系完成闭环。

现状基础（已具备）
：
1. 表结构已可承载证据与结果：
   - `CheckinSubmission`：`photoKey/photoUri/faceVerifyScore/faceVerifyPassed`。
   - `CheckinTask`：`faceRequired/faceMinScore`。
   - `FaceEmbedding`：模板存储能力已存在。
2. 上传能力已存在：
   - `POST /api/checkin/photos/upload` 已支持写入 R2 `records/`。
3. 审核链路已存在：
   - `autoResult/finalResult/manualResult` 可复用，无需新增审核体系。

分阶段执行计划（详细）
：
P0（协议冻结与开关定义）
1. 冻结签到提交契约：
   - 请求：`taskId/studentId + 业务字段 + photoKey/photoUri`；
   - 响应：统一返回 `submissionId/autoResult/finalResult/faceVerifyScore/faceVerifyPassed`。
2. 冻结任务策略：
   - `faceRequired=1` 时必须上传照片证据；
   - `faceMinScore` 为唯一阈值来源，禁止端侧私有阈值。
3. 冻结兼容策略：
   - 老客户端未上传证据时，按任务策略进入 `PENDING_REVIEW`，不直接崩溃拒绝。

P1（Worker 服务分层）
1. 路由层保持薄编排，仅做鉴权、参数归一化、错误映射。
2. 新增/完善服务层职责：
   - `faceEmbeddingService`：模板增删改查与激活版本；
   - `faceEnrollService`：模板注册与质量评估；
   - `faceVerifyService`：证据下载、特征提取、1:N 比对；
   - `facePolicyService`：阈值决策与 `autoResult/finalResult` 映射。
3. 写库规则统一：
   - 通过：`autoResult='PASS'`；
   - 不通过：`autoResult='FAIL'`，`finalResult` 按策略入 `PENDING_REVIEW/REJECTED`。

P2（签到主链路云端化）
1. `submitCheckin` 收到 `photoKey/photoUri` 后触发云端比对。
2. 缺证据与异常分类：
   - 缺证据、无脸、多人脸、低清晰度、模板缺失、服务超时分别返回可识别错误码。
3. 结果落库与透传：
   - `CheckinSubmission` 持久化分数与通过标记；
   - 任务详情与“我的提交”接口确保可直接展示证据与分数。

P3（观测、灰度与回滚）
1. 观测指标：
   - 上传成功率、比对耗时P95、低分率、审核回退率、申诉率。
2. 灰度顺序：
   - 单班级 -> 多班级 -> 全量班级；
   - 每阶段至少观察 3~7 天。
3. 回滚策略：
   - 任务级开关可一键回退到“仅证据收集+人工审核”模式；
   - 保留证据与日志，避免数据丢失。

接口与数据口径约束（执行前确认）
：
1. 时间字段统一 ISO（或明确 unixepoch），禁止混入未声明格式。
2. `faceVerifyScore` 统一 `0~1` 浮点；`faceVerifyPassed` 统一 `0/1`。
3. R2 路径固定：`records/{taskId}/{studentId}/{timestamp}_{uuid}.jpg`。
4. 图片访问默认私有，教师侧通过签名/代理读取，不开放公网直链。

测试与验收计划
：
1. 单元测试：
   - 阈值边界、异常分类、判定映射、字段归一化。
2. 集成测试：
   - 上传->提交->比对->写库->查询展示 全链路；
   - 并发提交与重复提交（`isLatest`）一致性。
3. 业务验收：
   - 人脸必选任务可闭环；
   - 审核端可见证据与分数；
   - 老版本客户端可兼容运行。

跨仓协同计划（FaceCheck + omniattend-core）
：
1. 先云端：完成 Worker 分层与返回契约稳定。
2. 后客户端：FaceCheck 接入“采集压缩上传+提交证据键”并灰度。
3. 最后收口：统一文档、监控、告警与回滚预案。

本轮说明
：
1. 本节仅为迁移规划，不包含代码改动。
2. 后续每个阶段落地后单独追加“完成记录 + 构建验证”。

4.6 规划：人脸识别全面上云（Android 仅压缩+上传）
omniattend-core

目标
：
1. Android 端不再做人脸特征提取与识别，仅负责照片压缩、上传与提交基础签到参数。
2. 云端 Worker 负责学生特征向量初始化、签到时比对、结果入库与异常进入人工批复队列。
3. 确保不影响现有 checkin 任务流（定位/手势/密码/人工审核）与既有 API 行为。

现有数据库概况（基于 schema.sql）
：
1. `FaceEmbedding` 已具备学生向量存储能力（`studentId/modelVer/vector/quality/isActive`），可直接承接“特征模板库”。
2. `CheckinSubmission` 已具备自动/人工结果链路（`autoResult/finalResult/manualResult/reviewerId/reviewedAt/reason`），可直接承接“人脸失败进审核队列”。
3. R2 已绑定 `omni` 桶，且已有上传接口，适合按 `records/` 前缀存签到照片证据。

最小侵入落地方案（不破坏旧链路）
：
P0（先落地）
1. 约定签到照片统一存储路径：`records/{taskId}/{studentId}/{timestamp}_{uuid}.jpg`。
2. Android 调整为：
   - 本地压缩图片（控制尺寸/质量）；
   - 调用现有上传接口上传到 R2；
   - 提交签到时仅传 `photoKey/photoUri` 与原有定位/手势/密码字段。
3. Worker 新增字段接收（兼容旧请求，字段可选）：
   - `CheckinSubmission` 增加 `photoKey TEXT`、`photoUri TEXT`（迁移脚本增量添加，避免重建表）。

P1（云端人脸能力接入）
1. 新增 Worker 路由（建议）：
   - `POST /api/face/enroll`：为学生初始化/更新特征向量（可多张图聚合）。
   - `POST /api/face/verify`：签到时按图片做比对并返回分数与建议结果。
2. 服务分层：
   - `services/face/faceEmbeddingService.ts`（已开始）
   - `services/face/faceEnrollService.ts`
   - `services/face/faceVerifyService.ts`
3. 结果写回策略：
   - 比对通过：`autoResult='PASS'`；
   - 比对失败或低置信：`autoResult='FAIL' + finalResult='PENDING_REVIEW'`，进入现有批复队列。

P2（审核增强，复用现有队列）
1. 审核端展示 `photoUri/photoKey + 比对分数 + 失败原因`。
2. 教师在现有 `/review-queue` 与 `/review` 机制中完成批复，不新增独立审核体系。

数据与安全约束
：
1. `FaceEmbedding.vector` 建议统一为可迁移编码（建议 `FLOAT32` 序列化后 base64 或 BLOB 二进制），保留 `modelVer` 做版本隔离。
2. R2 `records/` 建议启用最小可见范围（仅服务端签名访问），客户端不直连公开读。
3. 旧人脸接口暂不删除，先走兼容期；完成新链路灰度后再下线旧分支。

不影响既有业务的保证

1. 现有定位/手势/密码校验逻辑保持不变；
2. 仅新增可选字段与新路由，不改旧接口必填；
3. 审核继续复用 `CheckinSubmission` 的既有人工批复流程。
